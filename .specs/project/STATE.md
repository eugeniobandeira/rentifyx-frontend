Key decisions: proactive+reactive token refresh (decode `exp`, timer ~60s before expiry, interceptor 401 fallback); route guard redirects to `/login?returnUrl=`; post-register shows a local "check your email" state (no route, no resend — endpoint doesn't exist); no Okta/OAuth reuse from `estrutura.md` (contract is first-party JWT only); auth pages live in `core/pages/*` (register/login/verify-email/forgot-password/reset-password), authenticated profile+LGPD page lives in `features/identity/user/pages/account`; `BaseHttpService`/shared UI kit deliberately NOT built yet (single caller, avoids premature abstraction — revisit when a 2nd feature needs them). Full test suite: 87/88 passing (only failure is the pre-existing unrelated `app.spec.ts` issue, see Known Issues), production build green (client + server bundles, 7 prerendered routes).

Documented in `CLAUDE.md`. Applied retroactively to pre-identity code (`ThemeService`, `category-color.map.ts`) and throughout the identity feature.
- **Token storage — RESOLVED, superseded** (2026-07-11): `api-contracts.md` was revised — `refreshToken` is now a server-managed `httpOnly` cookie, never in a JSON body; `accessToken` is contract-mandated memory-only. This retroactively resolves the 2026-07-10 "leave it open" decision — there's no longer a real choice to make, the backend dictates it. New follow-on decision (confirmed with the user): persist only `email` (non-sensitive) in `localStorage` so `bootstrap()` can call `POST /auth/refresh` after a reload. Full plan: `.specs/features/identity/tasks.md` → "Migration: httpOnly Refresh-Token Cookie" (4 tasks, M1–M4). **Implemented and verified (2026-07-11)** — 91/92 tests passing (sole failure is the pre-existing unrelated `app.spec.ts` issue), production build green (7 prerendered routes). **Not yet committed** — per new "Git" policy in `CLAUDE.md`, commits only happen when explicitly requested.

- **State-management hardening (2026-07-11, planned)**: an audit of the implemented identity feature found 6 issues — duplicated current-user state (`SessionService.currentUser` vs `AccountPage`'s own local copy), duplicated access-token storage (`TokenStorageService._accessToken`, dead — `SessionService._accessTokenSignal` is the real owner), `Router` navigation buried inside `SessionService.refresh()`/`logout()` (couples state to routing, hard to unit test), copy-pasted error/view-state boilerplate across 4 auth page components (already diverged — `RegisterPage` handles a `pattern` case `ForgotPasswordPage` doesn't), inconsistent loading/error vocabulary (`AccountPage`'s `status: 'loading'|'loaded'|'error'` vs the other 4 pages' `submitting`/`banner`/`isRateLimit`), and a real (if low-impact) memory leak in `ThemeService._watchSystemPreference()` — a `matchMedia` `'change'` listener registered via `addEventListener` with no matching `removeEventListener`/`DestroyRef` cleanup. Also noted in passing: `SessionService.logout()` currently has zero callers anywhere in the app (no logout button wired yet) — out of scope, flagged only. Full plan: `.specs/features/identity/tasks.md` → "Hardening: State Ownership, Composable Extraction & Cleanup" (H1–H12), traced as new requirement IDENT-09 in `spec.md`. Key decisions: `SessionService` becomes the sole owner of token + current-user state (`TokenStorageService` persists only `email`); navigation-on-session-end moves out of `SessionService` into an `effect()` in `App`'s constructor watching `isAuthenticated()` for a `true→false` transition (first real use of Angular `effect()` in this codebase, paired with the first use of `DestroyRef` for the `ThemeService` fix); shared error/view-state logic across the 4 auth pages + `AccountPage` extracted into `shared/composables/use-form-submission.ts` (first file in a `composables/` folder — CLAUDE.md documented this convention aspirationally before anything used it). **Implemented and verified 2026-07-11** (H1–H12) — full suite 125/125 passing (the previously-known unrelated `app.spec.ts` failure is gone, superseded by an earlier router-outlet-based test), `npm run build` green (7 prerendered routes, client+server bundles), `npm run lint` green with no new violations. `TokenStorageService.setSession(accessToken, email)` was narrowed to `setEmail(email)` during implementation (accessToken param was dead) — minor naming deviation from the plan's literal wording, not from its intent. `CLAUDE.md` updated to describe the landed state (composable, session ownership, `Router`-free state services + `effect()`-based redirect, `DestroyRef` listener cleanup) as current fact, not "planned". **Not yet committed** — per `CLAUDE.md`'s Git policy (commits only on explicit request).
- **Structure alignment with estrutura.md — reorg (2026-07-10, superseding the earlier "auth pages live in core/pages/*" decision)**: user flagged 4 deviations from the blueprint after reviewing the codebase. All fixed:
  1. **Domain pages moved out of `core/pages/`, then split one-entity-per-flow (2026-07-10, follow-up round)**: initial pass moved all 5 pages to a shared `features/identity/auth/pages/`+`services/`+`interfaces/` — user pushed back: that's still a dumping ground, not entity boundaries. Re-split so each auth flow is its own entity under `features/identity/auth/`: `login/`, `register/` (+ `constants/register.constants.ts`), `verify-email/`, `forgot-password/` (+ `constants/forgot-password.constants.ts`), `reset-password/` (+ `constants/reset-password.constants.ts`) — each with its own `components/`(page)+`interfaces/`+`services/`. The monolithic `AuthService` (7 methods, one per endpoint) was split into one service per entity (`LoginService.login()`, `RegisterService.register()`, `VerifyEmailService.verifyEmail()`, `ForgotPasswordService.forgotPassword()`, `ResetPasswordService.resetPassword()`), each a single-method `HttpClient` wrapper. `session/` became its own entity too (no page — owns session lifecycle): `SessionService` (state/signals, unchanged logic) now injects `LoginService` (for `login()`) + new `SessionApiService` (`refresh()`/`logout()`, the two endpoints with no owning page) instead of the old `AuthService`; `TokenStorageService` also lives here. `iAuthTokenResponse`/`iRefreshRequest`/`iLogoutRequest` moved to `session/interfaces/` (session is the shape's real owner — login's response is a session being issued). Only `home` (and future `not-found`/`access-denied`) stay in `core/pages/` — those have no domain owner at all.
  2. *(superseded by 1 above — services now live per-entity, not in one shared `auth/services/`)*
  3. **Inline component templates extracted**: all 5 auth pages had 50-160 line `template:` strings inline in the `.ts`; each now has its own `.html` via `templateUrl`, matching every component in estrutura.md.
  4. **HTTP error classification centralized in an interceptor**: `login`/`register`/`forgot-password`/`reset-password` each repeated an `instanceof HttpErrorResponse` + `status === 401/422/429/0/...` if/else chain to pick a banner message. New `core/interceptors/http-error.interceptor.ts` catches every `HttpErrorResponse`, maps it to a `HttpErrorKind` (`shared/types/http-error-kind.ts` — first use of a new `types/` folder, sibling to `interfaces/`, for type aliases vs. object-shape interfaces), and rethrows `iClassifiedHttpError` (`shared/interfaces/classified-http-error.ts`). Pages now `switch`/compare on `.kind` instead of re-deriving status/instanceof logic. Registered as `[httpErrorInterceptor, authInterceptor]` in `app.config.ts` — classifier outermost so `authInterceptor`'s 401-refresh retry still sees the raw `HttpErrorResponse`. `CLAUDE.md` updated with all 4 conventions. **Verification pending** — see Todos.

Full contract summary in `.specs/codebase/INTEGRATIONS.md`. Found: (1) identity-api's contract mostly matches what this frontend already assumes (httpOnly refresh cookie, Bearer JWT body, `/api/v1` prefix) — CORS on that backend currently only allows `http://localhost:3000`, confirm the frontend's actual origin is added before live integration testing; (2) `iUserResponse`/`iDataExportResponse` silently dropped all consent fields the backend returns, and no `GET/PUT /users/me/consent` service existed — tracked as new requirement `IDENT-10`, task breakdown "Consent Alignment: Contract Fixes & Consent Management UI" (C1–C6) in `.specs/features/identity/tasks.md`; (3) `communications-api` is deliberately **not** integrated directly this cycle — user decision (confirmed via question, not a default): its endpoints require a static `X-Api-Key` unsafe to embed in an Angular SPA (explicitly "no identity-api/JWT coupling" per that repo's own AD-011), it has no HTTP endpoint to send a notification (Kafka-only intake) and no real-time channel at all (no SignalR/WebSocket), and its per-channel consent model is a different concept from identity-api's per-purpose consent handled by IDENT-10. Revisit only if a BFF/proxy is built to hold that key server-side. Design phase was skipped for IDENT-10 (Medium scope — reuses existing entity/service/composable conventions, no new architecture needed); went straight from spec to a 6-task breakdown.
