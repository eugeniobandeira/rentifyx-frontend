# State

## Decisions
- Use Angular standalone app architecture.
- Keep the initial structure aligned with the provided blueprint.
- Color system (2026-07-09): trust/security blue brand, orange accent reserved for CTAs, WCAG AA target, first-class dark mode via class-based Tailwind v4 `@custom-variant`, dedicated rental-status tokens (available/reserved/rented/unavailable/overdue), and color-per-category badges using a fixed 8-slot palette + round-robin fallback (accepted scalability trade-off). Full spec/context/design in `.specs/features/color-system/`.
- Identity feature (2026-07-09 planned, 2026-07-10 implemented): full spec + design + 21-task breakdown for the auth/session/LGPD surface of `api-contracts.md` (register, verify-email, login, refresh, logout, forgot/reset password, profile, data export/delete). **All 21 tasks implemented and committed** (T1–T21, 8 commits on `master`). Key decisions: proactive+reactive token refresh (decode `exp`, timer ~60s before expiry, interceptor 401 fallback); route guard redirects to `/login?returnUrl=`; post-register shows a local "check your email" state (no route, no resend — endpoint doesn't exist); no Okta/OAuth reuse from `estrutura.md` (contract is first-party JWT only); auth pages live in `core/pages/*` (register/login/verify-email/forgot-password/reset-password), authenticated profile+LGPD page lives in `features/identity/user/pages/account`; `BaseHttpService`/shared UI kit deliberately NOT built yet (single caller, avoids premature abstraction — revisit when a 2nd feature needs them). Full test suite: 87/88 passing (only failure is the pre-existing unrelated `app.spec.ts` issue, see Known Issues), production build green (client + server bundles, 7 prerendered routes).
- **Naming convention (2026-07-10)**: every `interface` gets a lowercase `i` prefix (`iUserResponse`, not `UserResponse`); every private class field/method gets a leading `_` prefix (`private readonly _http`). Documented in `CLAUDE.md`. Applied retroactively to pre-identity code (`ThemeService`, `category-color.map.ts`) and throughout the identity feature.
- **Token storage — RESOLVED, superseded** (2026-07-11): `api-contracts.md` was revised — `refreshToken` is now a server-managed `httpOnly` cookie, never in a JSON body; `accessToken` is contract-mandated memory-only. This retroactively resolves the 2026-07-10 "leave it open" decision — there's no longer a real choice to make, the backend dictates it. New follow-on decision (confirmed with the user): persist only `email` (non-sensitive) in `localStorage` so `bootstrap()` can call `POST /auth/refresh` after a reload. Full plan: `.specs/features/identity/tasks.md` → "Migration: httpOnly Refresh-Token Cookie" (4 tasks, M1–M4). **Implemented and verified (2026-07-11)** — 91/92 tests passing (sole failure is the pre-existing unrelated `app.spec.ts` issue), production build green (7 prerendered routes). **Not yet committed** — per new "Git" policy in `CLAUDE.md`, commits only happen when explicitly requested.
- **Git policy (2026-07-11)**: `CLAUDE.md` now has a "## Git" section — never run `git commit`/`git push` unless the user explicitly asks in that turn, even after a task/plan was approved for implementation.
- **Structure alignment with estrutura.md — reorg (2026-07-10, superseding the earlier "auth pages live in core/pages/*" decision)**: user flagged 4 deviations from the blueprint after reviewing the codebase. All fixed:
  1. **Domain pages moved out of `core/pages/`, then split one-entity-per-flow (2026-07-10, follow-up round)**: initial pass moved all 5 pages to a shared `features/identity/auth/pages/`+`services/`+`interfaces/` — user pushed back: that's still a dumping ground, not entity boundaries. Re-split so each auth flow is its own entity under `features/identity/auth/`: `login/`, `register/` (+ `constants/register.constants.ts`), `verify-email/`, `forgot-password/` (+ `constants/forgot-password.constants.ts`), `reset-password/` (+ `constants/reset-password.constants.ts`) — each with its own `components/`(page)+`interfaces/`+`services/`. The monolithic `AuthService` (7 methods, one per endpoint) was split into one service per entity (`LoginService.login()`, `RegisterService.register()`, `VerifyEmailService.verifyEmail()`, `ForgotPasswordService.forgotPassword()`, `ResetPasswordService.resetPassword()`), each a single-method `HttpClient` wrapper. `session/` became its own entity too (no page — owns session lifecycle): `SessionService` (state/signals, unchanged logic) now injects `LoginService` (for `login()`) + new `SessionApiService` (`refresh()`/`logout()`, the two endpoints with no owning page) instead of the old `AuthService`; `TokenStorageService` also lives here. `iAuthTokenResponse`/`iRefreshRequest`/`iLogoutRequest` moved to `session/interfaces/` (session is the shape's real owner — login's response is a session being issued). Only `home` (and future `not-found`/`access-denied`) stay in `core/pages/` — those have no domain owner at all.
  2. *(superseded by 1 above — services now live per-entity, not in one shared `auth/services/`)*
  3. **Inline component templates extracted**: all 5 auth pages had 50-160 line `template:` strings inline in the `.ts`; each now has its own `.html` via `templateUrl`, matching every component in estrutura.md.
  4. **HTTP error classification centralized in an interceptor**: `login`/`register`/`forgot-password`/`reset-password` each repeated an `instanceof HttpErrorResponse` + `status === 401/422/429/0/...` if/else chain to pick a banner message. New `core/interceptors/http-error-classifier.interceptor.ts` catches every `HttpErrorResponse`, maps it to a `HttpErrorKind` (`shared/types/http-error-kind.ts` — first use of a new `types/` folder, sibling to `interfaces/`, for type aliases vs. object-shape interfaces), and rethrows `iClassifiedHttpError` (`shared/interfaces/classified-http-error.ts`). Pages now `switch`/compare on `.kind` instead of re-deriving status/instanceof logic. Registered as `[httpErrorClassifierInterceptor, authInterceptor]` in `app.config.ts` — classifier outermost so `authInterceptor`'s 401-refresh retry still sees the raw `HttpErrorResponse`. `CLAUDE.md` updated with all 4 conventions. **Verification pending** — see Todos.
