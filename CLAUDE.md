# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project status

The identity feature (auth, session, LGPD data export/delete) is implemented under `src/app/features/identity/`, following the entity-per-domain structure documented below. `src/app/core` and `src/app/shared` hold cross-cutting infra and reusable pieces respectively — see the Folder convention below for what belongs where.

This repo no longer keeps `estrutura.md` (the original target folder-structure blueprint, copied from a larger sibling project) or `api-contracts.md` (the RentifyX Identity backend API contract) as files — both were reference material used to converge on the conventions below, which are now the authoritative source; several sections still cite "estrutura.md" by name for historical attribution (naming patterns like `*-form.config.ts`, `test-id.constants.ts`), not as a pointer to a readable file. For the backend contract (request/response shapes, validation rules, error formats), check the actual backend repo/OpenAPI spec if one is available — don't assume the shapes documented here are exhaustive.
- `.specs/project/` (`PROJECT.md`, `ROADMAP.md`, `STATE.md`) — check `STATE.md` for the latest decisions/todos before assuming later-phase architecture is in place.

## Commands

Package manager is npm (`packageManager: npm@11.16.0` in package.json).

```bash
npm start           # ng serve — dev server at http://localhost:4200
npm run build       # ng build (production by default)
npm run watch       # ng build --watch --configuration development
npm test            # ng test — runs unit tests via Vitest (Angular's unit-test builder)
```

- Run a single test file: `ng test --include='**/home.spec.ts'` (adjust path glob as needed).
- There is no e2e framework configured yet (`ng e2e` is a no-op stub). CI/CD is not set up yet either — lint/build/test are run locally for now.
- `npm run lint` (`ng lint`) runs ESLint (`@angular-eslint/schematics`, config in `eslint.config.js`), on top of Prettier for formatting (`.prettierrc`: single quotes, 100-char print width, Angular parser for `.html`). The lint config encodes conventions that used to be manual-only: `@typescript-eslint/no-explicit-any` (no `any` anywhere), `@typescript-eslint/naming-convention` enforcing the `i`-prefix on interfaces and the `_`-prefix on private class members (private `readonly` constants like `_API_URL` are allowed `UPPER_CASE` after the `_`, matching the URL-field convention below). `@typescript-eslint/consistent-type-definitions` is turned off because form-config files deliberately use `type <Name>FormControl` (see the reactive-forms convention below), not `interface`, for that internal control-name mapping. `@typescript-eslint/no-empty-function` is relaxed in `*.spec.ts` since empty mock implementations (e.g. stubbing `addEventListener`) are expected there.
- SSR: `npm run serve:ssr:rentityx-frontend` runs the built Express server (`dist/rentityx-frontend/server/server.mjs`) — only works after a production build.

## Architecture

- **Angular 22**, standalone-component style (no NgModules) — components declare `imports` directly, routes are plain `Routes` arrays.
- **Environment config**: `src/app/environment/environment.ts` exports a single `environment` object — `production: boolean` + `apiUrl` as a **full absolute URL** (`http://localhost:5000/api/v1` — confirm/update the port to match whatever the backend actually runs on locally), not a relative path. There is deliberately no `API_BASE_URL` re-export/constant wrapping it — don't reintroduce one. Never hardcode a relative `/api/v1/...` path in a service or test, always compose from `environment.apiUrl`. This is intentionally a single file for now (no `environment.development.ts`/`fileReplacements` split yet) — `api-contracts.md` calls for different base URLs per environment (local vs `https://{host}` production), so revisit this when that split is actually needed.
  - **Usage in `@Injectable` services**: don't import `environment` and build a module-level URL constant (`const X_URL = \`${environment.apiUrl}/...\`` outside the class). Instead build the full (or shared-prefix) resource URL as a private class field — `private readonly _API_URL = \`${environment.apiUrl}/auth/login\`;` — and reference `this._API_URL` inside each method (see `login.service.ts`, `user.service.ts`). For services with multiple endpoints under one prefix (e.g. `session-api.service.ts`'s `refresh`/`logout`), `_API_URL` holds just the shared prefix and each method appends its own suffix (`` `${this._API_URL}/refresh` ``). This keeps the URL an instance concern like `_http`, consistent with the `_`-prefixed private-member convention, rather than module-scope state.
  - **Usage in functional interceptors** (no class/`this` to hold a field): import `environment` and reference it directly, a module-level constant derived from it is fine there (see `auth.interceptor.ts`).
- **SSR is enabled** via `@angular/ssr`: `src/main.ts` (browser bootstrap), `src/main.server.ts` + `app.config.server.ts` (server bootstrap, merges `provideServerRendering()` into the client `appConfig`), and `src/server.ts` (Express server handling SSR + static assets). `app.routes.server.ts` controls server-side render mode. `provideClientHydration()` is enabled in `app.config.ts` for hydration on the client.
- **Routing**: `src/app/app.routes.ts` is the root route table; wildcard paths redirect to `''`. As features are added, follow the blueprint's per-feature `*.routes.ts` pattern (e.g. `commodity-hub.routes.ts`) and lazy-load feature route files rather than growing the root routes file.
- **Folder convention** (originally converged from the now-removed `estrutura.md` blueprint, applied as features are built):
  - `core/` — cross-cutting infra with **no domain ownership**: guards, interceptors, the layout shell, and only pages that don't belong to any feature domain (`home`, `not-found`, `access-denied`). A page or service tied to a specific domain (e.g. the auth flow, a user's profile) belongs under that feature, not here, even if it's part of the unauthenticated shell. `core/interceptors/` and `core/guards/` are never a flat bag of files either — each interceptor/guard gets its own subfolder named after it (e.g. `core/interceptors/auth/auth.interceptor.ts`+`.spec.ts`, `core/interceptors/http-error/http-error.interceptor.ts`+`.spec.ts`), one concern per folder, same as entities under `features/`.
  - `features/<domain>/<entity>/` — each **entity** (not just each domain) gets its own subfolder with its own `components/`, `constants/`, `interfaces/`, `types/`, `services/` (only the ones it actually needs — e.g. an entity with no local constant doesn't get an empty `constants/`). Don't group multiple unrelated entities' pages/services/interfaces into one shared `pages/`/`services/`/`interfaces/` folder at the domain root — that's a dumping ground, not an entity boundary. Example: `features/identity/auth/` has no files of its own; it's purely a parent for its entities — `login/` (`components/login.*`, `interfaces/login-request.ts`, `services/login.service.ts`), `register/` (adds `constants/register.constants.ts` for its password pattern + field-name list), `verify-email/`, `forgot-password/`, `reset-password/`, and `session/` (no page — owns session lifecycle: `services/session.service.ts` for state, `services/session-api.service.ts` for the refresh/logout HTTP calls, `services/token-storage.service.ts`, `interfaces/auth-token-response.ts`+`refresh-request.ts`+`logout-request.ts`). Request/response DTOs live in each entity's `interfaces/` as one file per shape (`create-x-request.ts`, `x-response.ts`, etc.), matching the backend's actual request/response shapes. A page belongs inside its owning entity, never in `core/pages/` — `core/pages/` is reserved for pages with no domain owner at all (`home`, `not-found`, `access-denied`).
  - `shared/` — cross-feature reusable pieces: generic `ui/` component kit (button, input, select, table, form-dialog, pagination...), `composables/` (reusable reactive logic — a plain function returning readonly signals/methods, callable as a component field initializer; e.g. `use-list-pagination.ts`, `use-form-field.ts`, and `use-form-submission.ts` — a submitting/banner/isRateLimit + field-error-message + server-validation-mapping toolkit shared across the identity feature's reactive-form pages, `LoginPage`/`RegisterPage`/`ForgotPasswordPage`/`ResetPasswordPage`/`AccountPage`; it's a toolkit, not a controller — a page whose error handling needs an extra branch, e.g. `ResetPasswordPage`'s `bad-request`/`not-found` → `invalid-link` redirect, checks that case itself first and falls through to the composable's `handleError` for everything else), `services/` (e.g. `base-http.service.ts` for common HTTP concerns), `directives/`, `constants/`, `interfaces/` (e.g. `list-api-request.ts`/`list-api-response.ts` for paginated list endpoints), `types/` (union/primitive type aliases shared across features, e.g. `http-error-kind.ts`).
- **Imports**: use the path aliases defined in `tsconfig.json` instead of long relative paths — `@app/*`, `@core/*`, `@features/*`, `@shared/*` (all map into `src/app/...`). E.g. `import { BaseHttpService } from '@shared/services/base-http.service'` rather than `'../../../../shared/services/base-http.service'`. Relative imports (`./`, `../`) are still fine for files within the same feature/entity folder.
- **One interface/type per file**: never collect multiple interfaces or DTOs into a single file (no `models.ts` / `types.ts` grab-bags). Each request/response shape gets its own file, following the blueprint's `create-x-request.ts`, `update-x-request.ts`, `filter-x-request.ts`, `x-response.ts` naming — even for small interfaces. An `index.ts` barrel re-exporting a folder's files is fine when the blueprint already uses one (see `interfaces/index.ts`, `services/index.ts` patterns in estrutura.md), but the interfaces themselves still live in separate files.
- **`interfaces/` vs `types/`**: object-shape contracts (anything you'd write as `interface`, prefixed `i...`) go in `interfaces/`. Union/primitive/utility type aliases (anything you'd write as `type`, no `i` prefix, PascalCase — e.g. `HttpErrorKind`) go in a sibling `types/` folder. Don't mix the two in one file or one folder.
- **Components: template/style always in their own file** — never use an inline `template:`/`styles:` string in a `@Component` decorator once the markup is more than a couple of lines. Use `templateUrl: './x.html'` (and `styleUrl: './x.css'` if a stylesheet is needed) so `.ts`/`.html`/`.spec.ts` stay separate files, matching every component in estrutura.md.
- **Reactive forms — extract into a `constants/<name>-form.config.ts` factory, don't build the `FormGroup` inline in the component**: every entity with a reactive form gets a `constants/<entity>-form.config.ts` (per the `estrutura.md` `*-form.config.ts` naming) exporting:
  1. A private (non-exported) `<Name>FormControl` type — a plain object type mapping each control name to its `FormControl<T>`.
  2. A `create<Name>FormControl(): FormGroup<...>` function that `inject(NonNullableFormBuilder)`s and returns `fb.group({...})`, each field built with `new FormControl(default, { validators: [...], nonNullable: true })`.
  3. Two derived exported types: `<Name>FormGroup = ReturnType<typeof create<Name>FormControl>` and `<Name>FormValue = ReturnType<<Name>FormGroup['getRawValue']>`.
  See `login/constants/login-form.config.ts`, `register/constants/register-form.config.ts`, `forgot-password/constants/forgot-password-form.config.ts`, `reset-password/constants/reset-password-form.config.ts` for the pattern. The component then just does `readonly form: <Name>FormGroup = create<Name>FormControl();` as a field initializer (this is a valid Angular injection context, same as `private readonly _http = inject(HttpClient)`) — no `FormBuilder` injected or `Validators` imported directly in the component. If the form shares a validation pattern/constant with other parts of the entity (e.g. a password regex), import it from the entity's existing `<entity>.constants.ts` rather than duplicating it in the form config file.
- **Testability — every meaningful element gets a `data-testid`**: forms, inputs/selects/checkboxes, buttons, links, and conditional state containers (loading/success/error/banner views, `@switch`/`@if` branches a test needs to assert on) all carry a `data-testid` attribute — don't make tests/e2e select by CSS class, text content, or Tailwind utility classes, those change with styling and break tests for reasons unrelated to behavior.
  - **Format**: kebab-case, prefixed with the component's name (its selector minus the `app-` prefix) to avoid collisions across pages — e.g. `app-login` → `data-testid="login-email-input"`, `data-testid="login-submit-button"`, `data-testid="login-error-banner"`.
  - **Static vs dynamic**: a plain `data-testid="login-submit-button"` attribute for static elements; `[attr.data-testid]="..."` only when the id itself must vary (rare — prefer a stable id and vary content/state instead).
  - **Scope**: page/feature-specific testids are literal strings directly in the template (and referenced the same way in the `.spec.ts`) — no need to centralize every one. Reserve a shared `shared/constants/test-id.constants.ts` (per the `estrutura.md` blueprint) only for testids genuinely reused across multiple unrelated components (e.g. a shared `ui/` kit's generic loading-spinner or error-banner testid), so it doesn't become a second models.ts-style grab-bag.
- **HTTP error handling — classify in an interceptor, branch on `kind` in the caller**: `core/interceptors/http-error.interceptor.ts` catches every `HttpErrorResponse`, maps its status to a `HttpErrorKind` (`@shared/types/http-error-kind.ts`), and rethrows a normalized `iClassifiedHttpError` (`@shared/interfaces/classified-http-error.ts`) with a ready-to-display `message`. Callers (components, services) must not re-derive error meaning via `instanceof HttpErrorResponse` / `status === N` checks — they receive `iClassifiedHttpError` and switch on `.kind`. Registered outermost in `app.config.ts`'s interceptor chain (`[httpErrorInterceptor, authInterceptor]`) so `authInterceptor`'s 401-refresh logic still sees the raw `HttpErrorResponse` first.
- **Styling — Tailwind-first, mobile-first, responsive**: Tailwind CSS v4 via `@tailwindcss/postcss` (see `.postcssrc.json`); global import in `src/styles.css` is `@import 'tailwindcss';`. Prefer Tailwind utility classes in templates over new component CSS files — only add a `.css` file for styles Tailwind can't express (complex animations, third-party overrides). Component-level CSS files, when needed, sit alongside the component.
  - Write styles unprefixed (mobile) first, then layer up with `sm:`/`md:`/`lg:`/`xl:` for larger breakpoints — never design desktop-first and retrofit mobile.
  - Every screen/component must be usable and legible at narrow viewports (~360px) as well as desktop; avoid fixed pixel widths, prefer fluid/relative units (`%`, `rem`, `flex`, `grid`) and `max-w-*` containers.
  - Avoid `@apply`-heavy custom CSS classes that duplicate what utilities already give you; keep styling colocated and composable rather than introducing global CSS abstractions.
- **Performance and scalability**: keep this in mind for every non-trivial addition, not just styling.
  - Prefer standalone lazy-loaded feature routes (`loadChildren`/`loadComponent`) over eagerly importing every feature into the root route table, so the initial bundle stays small — watch the budgets already defined in `angular.json` (`initial` 500kB warning / 1MB error, `anyComponentStyle` 4kB warning / 8kB error).
  - Use Angular's `OnPush` change detection and signals for component state where practical; avoid unnecessary subscriptions/re-renders.
  - Avoid loading large third-party libraries for something a few utility classes or a small composable can do.
  - Design shared/`ui` components and `composables` to be generic and reusable across features (per the `estrutura.md` blueprint) rather than duplicating logic per feature — this is what keeps the app scalable as more `features/<domain>/<entity>` folders are added.
- **TypeScript strictness**: `strict: true` (the full umbrella flag — `strictNullChecks`, `noImplicitAny`, `strictPropertyInitialization`, etc.) is on in `tsconfig.json`, plus `noImplicitOverride`, `noPropertyAccessFromIndexSignature`, `noImplicitReturns`, `noFallthroughCasesInSwitch`, and Angular-specific `strictInjectionParameters`/`strictInputAccessModifiers` — write code compatible with these from the start rather than retrofitting.
- **Guards and async app state**: a `CanActivateFn` must not read signal-based app state (e.g. `SessionService.isAuthenticated()`) synchronously if that state can still be mid-update from an in-flight async operation — Angular resolves route guards independent of whether an outlet is currently rendered, so a synchronous check can race an async bootstrap (e.g. session restoration on page reload) and make a valid session look logged-out. Where a service exposes both a result signal and an in-progress signal (e.g. `SessionService.isAuthenticated` / `isRestoringSession`), the guard should convert the in-progress signal with `toObservable` and gate its decision (`filter`, `take(1)`) on it settling first, returning an `Observable<boolean | UrlTree>` — see `core/guards/auth/auth.guard.ts`.
- **Session state ownership — `SessionService` is the single source of truth**: `SessionService` alone owns the access token (`_accessTokenSignal`) and the current user (`_currentUserSignal`, exposed as `currentUser`). `TokenStorageService` persists only `email` (needed to re-attempt `POST /auth/refresh` on reload) — it must never hold a second copy of the token. A component that fetches fresher user data (e.g. `AccountPage` calling `UserService.getMe()`) pushes the result back into `SessionService` via `updateCurrentUser()` rather than keeping its own local `user` signal — read `SessionService.currentUser` for display instead. Don't reintroduce a second, independently-fetched copy of session-owned data anywhere else.
- **State services stay routing-free — no `Router` calls inside state-mutation methods**: a service that owns app state (like `SessionService`) must not call `Router.navigateByUrl()`/`navigate()` from within its own state-changing methods (e.g. `refresh()`, `logout()`) — it makes the service impossible to unit-test without mocking `Router`, and hides a navigation side effect inside what looks like a pure state change. Instead, react to the resulting state change at a composition root: e.g. `App`'s constructor holds an `effect()` watching `SessionService.isAuthenticated()` for a `true → false` transition and navigates from there. This is also the codebase's reference pattern for using `effect()` to react to a signal change that crosses service/component boundaries — prefer it over hand-wiring the same reaction at every call site.
- **Listener/subscription cleanup — use `DestroyRef`, don't leave native listeners dangling**: any service or component that calls a native `addEventListener` (e.g. `matchMedia(...).addEventListener('change', ...)`) must register the matching `removeEventListener` via `inject(DestroyRef).onDestroy(...)` — keep the handler a named function so the same reference can be passed to both calls. `providedIn: 'root'` services only pay this cost once per app lifetime, but the convention exists so it isn't skipped on a component-scoped service or a future long-lived subscription where it would actually leak. See `core/services/theme.service.ts`'s `_watchSystemPreference()` for the reference implementation.
- **Naming conventions**:
  - Every `interface` name starts with a lowercase `i` prefix (e.g. `iUser`, `iAuthRequest`, `iListApiResponse`) — not uppercase `I`.
  - Every private class member (fields and methods) is prefixed with `_` (e.g. `private _tokenStorage`, `private _handleError()`).
  - Prefer precise, explicit types everywhere; avoid `any` — use `unknown` with narrowing, generics, or a proper interface/type instead.
- **Testing**: uses Angular's Vitest-based unit-test builder (`@angular/build:unit-test`), not Karma/Jasmine runner config. Test files are colocated as `*.spec.ts` next to the code under test.
