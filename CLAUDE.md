# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project status

The identity feature (auth, session, LGPD data export/delete) is implemented under `src/app/features/identity/`, following the entity-per-domain structure documented below. `src/app/core` and `src/app/shared` hold cross-cutting infra and reusable pieces respectively тАФ see the Folder convention below for what belongs where.

This repo no longer keeps `estrutura.md` (the original target folder-structure blueprint, copied from a larger sibling project) or `api-contracts.md` (the RentifyX Identity backend API contract) as files тАФ both were reference material used to converge on the conventions below, which are now the authoritative source; several sections still cite "estrutura.md" by name for historical attribution (naming patterns like `*-form.config.ts`, `test-id.constants.ts`), not as a pointer to a readable file. For the backend contract (request/response shapes, validation rules, error formats), check the actual backend repo/OpenAPI spec if one is available тАФ don't assume the shapes documented here are exhaustive.
- `.specs/project/` (`PROJECT.md`, `ROADMAP.md`, `STATE.md`) тАФ check `STATE.md` for the latest decisions/todos before assuming later-phase architecture is in place.

## Commands

Package manager is npm (`packageManager: npm@11.16.0` in package.json).

```bash
npm start           # ng serve тАФ dev server at http://localhost:4200
npm run build       # ng build (production by default)
npm run watch       # ng build --watch --configuration development
npm test            # ng test тАФ runs unit tests via Vitest (Angular's unit-test builder)
```

- Run a single test file: `ng test --include='**/home.spec.ts'` (adjust path glob as needed).
- There is no e2e framework configured yet (`ng e2e` is a no-op stub).
- No standalone lint script is defined in package.json; formatting is enforced via Prettier (`.prettierrc`: single quotes, 100-char print width, Angular parser for `.html`).
- SSR: `npm run serve:ssr:rentityx-frontend` runs the built Express server (`dist/rentityx-frontend/server/server.mjs`) тАФ only works after a production build.

## Architecture

- **Angular 22**, standalone-component style (no NgModules) тАФ components declare `imports` directly, routes are plain `Routes` arrays.
- **Environment config**: `src/app/environment/environment.ts` exports a single `environment` object тАФ `production: boolean` + `apiUrl` as a **full absolute URL** (`http://localhost:5000/api/v1` тАФ confirm/update the port to match whatever the backend actually runs on locally), not a relative path. There is deliberately no `API_BASE_URL` re-export/constant wrapping it тАФ don't reintroduce one. Never hardcode a relative `/api/v1/...` path in a service or test, always compose from `environment.apiUrl`. This is intentionally a single file for now (no `environment.development.ts`/`fileReplacements` split yet) тАФ `api-contracts.md` calls for different base URLs per environment (local vs `https://{host}` production), so revisit this when that split is actually needed.
  - **Usage in `@Injectable` services**: don't import `environment` and build a module-level URL constant (`const X_URL = \`${environment.apiUrl}/...\`` outside the class). Instead build the full (or shared-prefix) resource URL as a private class field тАФ `private readonly _API_URL = \`${environment.apiUrl}/auth/login\`;` тАФ and reference `this._API_URL` inside each method (see `login.service.ts`, `user.service.ts`). For services with multiple endpoints under one prefix (e.g. `session-api.service.ts`'s `refresh`/`logout`), `_API_URL` holds just the shared prefix and each method appends its own suffix (`` `${this._API_URL}/refresh` ``). This keeps the URL an instance concern like `_http`, consistent with the `_`-prefixed private-member convention, rather than module-scope state.
  - **Usage in functional interceptors** (no class/`this` to hold a field): import `environment` and reference it directly, a module-level constant derived from it is fine there (see `auth.interceptor.ts`).
- **SSR is enabled** via `@angular/ssr`: `src/main.ts` (browser bootstrap), `src/main.server.ts` + `app.config.server.ts` (server bootstrap, merges `provideServerRendering()` into the client `appConfig`), and `src/server.ts` (Express server handling SSR + static assets). `app.routes.server.ts` controls server-side render mode. `provideClientHydration()` is enabled in `app.config.ts` for hydration on the client.
- **Routing**: `src/app/app.routes.ts` is the root route table; wildcard paths redirect to `''`. As features are added, follow the blueprint's per-feature `*.routes.ts` pattern (e.g. `commodity-hub.routes.ts`) and lazy-load feature route files rather than growing the root routes file.
- **Folder convention** (originally converged from the now-removed `estrutura.md` blueprint, applied as features are built):
  - `core/` тАФ cross-cutting infra with **no domain ownership**: guards, interceptors, the layout shell, and only pages that don't belong to any feature domain (`home`, `not-found`, `access-denied`). A page or service tied to a specific domain (e.g. the auth flow, a user's profile) belongs under that feature, not here, even if it's part of the unauthenticated shell. `core/interceptors/` and `core/guards/` are never a flat bag of files either тАФ each interceptor/guard gets its own subfolder named after it (e.g. `core/interceptors/auth/auth.interceptor.ts`+`.spec.ts`, `core/interceptors/http-error/http-error.interceptor.ts`+`.spec.ts`), one concern per folder, same as entities under `features/`.
  - `features/<domain>/<entity>/` тАФ each **entity** (not just each domain) gets its own subfolder with its own `components/`, `constants/`, `interfaces/`, `types/`, `services/` (only the ones it actually needs тАФ e.g. an entity with no local constant doesn't get an empty `constants/`). Don't group multiple unrelated entities' pages/services/interfaces into one shared `pages/`/`services/`/`interfaces/` folder at the domain root тАФ that's a dumping ground, not an entity boundary. Example: `features/identity/auth/` has no files of its own; it's purely a parent for its entities тАФ `login/` (`components/login.*`, `interfaces/login-request.ts`, `services/login.service.ts`), `register/` (adds `constants/register.constants.ts` for its password pattern + field-name list), `verify-email/`, `forgot-password/`, `reset-password/`, and `session/` (no page тАФ owns session lifecycle: `services/session.service.ts` for state, `services/session-api.service.ts` for the refresh/logout HTTP calls, `services/token-storage.service.ts`, `interfaces/auth-token-response.ts`+`refresh-request.ts`+`logout-request.ts`). Request/response DTOs live in each entity's `interfaces/` as one file per shape (`create-x-request.ts`, `x-response.ts`, etc.), matching the backend's actual request/response shapes. A page belongs inside its owning entity, never in `core/pages/` тАФ `core/pages/` is reserved for pages with no domain owner at all (`home`, `not-found`, `access-denied`).
  - `shared/` тАФ cross-feature reusable pieces: generic `ui/` component kit (button, input, select, table, form-dialog, pagination...), `composables/` (reusable reactive logic, e.g. `use-list-pagination.ts`, `use-form-field.ts`), `services/` (e.g. `base-http.service.ts` for common HTTP concerns), `directives/`, `constants/`, `interfaces/` (e.g. `list-api-request.ts`/`list-api-response.ts` for paginated list endpoints), `types/` (union/primitive type aliases shared across features, e.g. `http-error-kind.ts`).
- **Imports**: use the path aliases defined in `tsconfig.json` instead of long relative paths тАФ `@app/*`, `@core/*`, `@features/*`, `@shared/*` (all map into `src/app/...`). E.g. `import { BaseHttpService } from '@shared/services/base-http.service'` rather than `'../../../../shared/services/base-http.service'`. Relative imports (`./`, `../`) are still fine for files within the same feature/entity folder.
- **One interface/type per file**: never collect multiple interfaces or DTOs into a single file (no `models.ts` / `types.ts` grab-bags). Each request/response shape gets its own file, following the blueprint's `create-x-request.ts`, `update-x-request.ts`, `filter-x-request.ts`, `x-response.ts` naming тАФ even for small interfaces. An `index.ts` barrel re-exporting a folder's files is fine when the blueprint already uses one (see `interfaces/index.ts`, `services/index.ts` patterns in estrutura.md), but the interfaces themselves still live in separate files.
- **`interfaces/` vs `types/`**: object-shape contracts (anything you'd write as `interface`, prefixed `i...`) go in `interfaces/`. Union/primitive/utility type aliases (anything you'd write as `type`, no `i` prefix, PascalCase тАФ e.g. `HttpErrorKind`) go in a sibling `types/` folder. Don't mix the two in one file or one folder.
- **Components: template/style always in their own file** тАФ never use an inline `template:`/`styles:` string in a `@Component` decorator once the markup is more than a couple of lines. Use `templateUrl: './x.html'` (and `styleUrl: './x.css'` if a stylesheet is needed) so `.ts`/`.html`/`.spec.ts` stay separate files, matching every component in estrutura.md.
- **Reactive forms тАФ extract into a `constants/<name>-form.config.ts` factory, don't build the `FormGroup` inline in the component**: every entity with a reactive form gets a `constants/<entity>-form.config.ts` (per the `estrutura.md` `*-form.config.ts` naming) exporting:
  1. A private (non-exported) `<Name>FormControl` type тАФ a plain object type mapping each control name to its `FormControl<T>`.
  2. A `create<Name>FormControl(): FormGroup<...>` function that `inject(NonNullableFormBuilder)`s and returns `fb.group({...})`, each field built with `new FormControl(default, { validators: [...], nonNullable: true })`.
  3. Two derived exported types: `<Name>FormGroup = ReturnType<typeof create<Name>FormControl>` and `<Name>FormValue = ReturnType<<Name>FormGroup['getRawValue']>`.
  See `login/constants/login-form.config.ts`, `register/constants/register-form.config.ts`, `forgot-password/constants/forgot-password-form.config.ts`, `reset-password/constants/reset-password-form.config.ts` for the pattern. The component then just does `readonly form: <Name>FormGroup = create<Name>FormControl();` as a field initializer (this is a valid Angular injection context, same as `private readonly _http = inject(HttpClient)`) тАФ no `FormBuilder` injected or `Validators` imported directly in the component. If the form shares a validation pattern/constant with other parts of the entity (e.g. a password regex), import it from the entity's existing `<entity>.constants.ts` rather than duplicating it in the form config file.
- **Testability тАФ every meaningful element gets a `data-testid`**: forms, inputs/selects/checkboxes, buttons, links, and conditional state containers (loading/success/error/banner views, `@switch`/`@if` branches a test needs to assert on) all carry a `data-testid` attribute тАФ don't make tests/e2e select by CSS class, text content, or Tailwind utility classes, those change with styling and break tests for reasons unrelated to behavior.
  - **Format**: kebab-case, prefixed with the component's name (its selector minus the `app-` prefix) to avoid collisions across pages тАФ e.g. `app-login` тЖТ `data-testid="login-email-input"`, `data-testid="login-submit-button"`, `data-testid="login-error-banner"`.
  - **Static vs dynamic**: a plain `data-testid="login-submit-button"` attribute for static elements; `[attr.data-testid]="..."` only when the id itself must vary (rare тАФ prefer a stable id and vary content/state instead).
  - **Scope**: page/feature-specific testids are literal strings directly in the template (and referenced the same way in the `.spec.ts`) тАФ no need to centralize every one. Reserve a shared `shared/constants/test-id.constants.ts` (per the `estrutura.md` blueprint) only for testids genuinely reused across multiple unrelated components (e.g. a shared `ui/` kit's generic loading-spinner or error-banner testid), so it doesn't become a second models.ts-style grab-bag.
- **HTTP error handling тАФ classify in an interceptor, branch on `kind` in the caller**: `core/interceptors/http-error.interceptor.ts` catches every `HttpErrorResponse`, maps its status to a `HttpErrorKind` (`@shared/types/http-error-kind.ts`), and rethrows a normalized `iClassifiedHttpError` (`@shared/interfaces/classified-http-error.ts`) with a ready-to-display `message`. Callers (components, services) must not re-derive error meaning via `instanceof HttpErrorResponse` / `status === N` checks тАФ they receive `iClassifiedHttpError` and switch on `.kind`. Registered outermost in `app.config.ts`'s interceptor chain (`[httpErrorInterceptor, authInterceptor]`) so `authInterceptor`'s 401-refresh logic still sees the raw `HttpErrorResponse` first.
- **Styling тАФ Tailwind-first, mobile-first, responsive**: Tailwind CSS v4 via `@tailwindcss/postcss` (see `.postcssrc.json`); global import in `src/styles.css` is `@import 'tailwindcss';`. Prefer Tailwind utility classes in templates over new component CSS files тАФ only add a `.css` file for styles Tailwind can't express (complex animations, third-party overrides). Component-level CSS files, when needed, sit alongside the component.
  - Write styles unprefixed (mobile) first, then layer up with `sm:`/`md:`/`lg:`/`xl:` for larger breakpoints тАФ never design desktop-first and retrofit mobile.
  - Every screen/component must be usable and legible at narrow viewports (~360px) as well as desktop; avoid fixed pixel widths, prefer fluid/relative units (`%`, `rem`, `flex`, `grid`) and `max-w-*` containers.
  - Avoid `@apply`-heavy custom CSS classes that duplicate what utilities already give you; keep styling colocated and composable rather than introducing global CSS abstractions.
- **Performance and scalability**: keep this in mind for every non-trivial addition, not just styling.
  - Prefer standalone lazy-loaded feature routes (`loadChildren`/`loadComponent`) over eagerly importing every feature into the root route table, so the initial bundle stays small тАФ watch the budgets already defined in `angular.json` (`initial` 500kB warning / 1MB error, `anyComponentStyle` 4kB warning / 8kB error).
  - Use Angular's `OnPush` change detection and signals for component state where practical; avoid unnecessary subscriptions/re-renders.
  - Avoid loading large third-party libraries for something a few utility classes or a small composable can do.
  - Design shared/`ui` components and `composables` to be generic and reusable across features (per the `estrutura.md` blueprint) rather than duplicating logic per feature тАФ this is what keeps the app scalable as more `features/<domain>/<entity>` folders are added.
- **TypeScript strictness**: `noImplicitOverride`, `noPropertyAccessFromIndexSignature`, `noImplicitReturns`, `noFallthroughCasesInSwitch`, plus Angular-specific `strictInjectionParameters` and `strictInputAccessModifiers` are all on тАФ write code compatible with these from the start rather than retrofitting.
- **Naming conventions**:
  - Every `interface` name starts with a lowercase `i` prefix (e.g. `iUser`, `iAuthRequest`, `iListApiResponse`) тАФ not uppercase `I`.
  - Every private class member (fields and methods) is prefixed with `_` (e.g. `private _tokenStorage`, `private _handleError()`).
  - Prefer precise, explicit types everywhere; avoid `any` тАФ use `unknown` with narrowing, generics, or a proper interface/type instead.
- **Testing**: uses Angular's Vitest-based unit-test builder (`@angular/build:unit-test`), not Karma/Jasmine runner config. Test files are colocated as `*.spec.ts` next to the code under test.
